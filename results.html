Parallelism(//-ism)
























Observations:











Moving from 1 processor to 2 processors almost doubles the performance.











The code used in this part is best parallelized on 2 processors. Increasing the no of processors above 2 increases the total time due to the extra overhead.











Maximum parallelization achieved with 2 processors.











The sequential Merge still causes a bottleneck in performance.











The efficiency increases as n increases, because there is more work for each processor to do---relatively less idle time.











Better values for slackness(>1); getting closer to linear speedup at n = 1M and p = 1











Tried parallelizing the merge function as well, but it has a bad effect on the overall performance(does no good) because it's not parallelizable(since it's a sequence of while loops without recursion), so it just causes an overhead to the scheduler.

Parallelism(//-ism)
























Observations:











The speedup is almost linear as it shows in the table :D











The code is highly- parallelizable especially with large n. Notice that //-ism values for n =1M and more are >>p and thus getting closer to linear speedup.











The work and total time are almost identical when only 1 processor is used(p=1); which means that sequential code is minimal











The fully-parallelized implementation is doing worse than the ½ parallelized version. It seems that, even though it has become more parallelizable due to spawning recursive merge calls, but the new recursive merge function is less efficient than the loop-based implementation.
